01System Overview & Query Lifecycle

Last verified: 2026-04-20 against commit a9d7ab3

PurposeWhat this system is

The SoS PoC is a retrieval-augmented question answering system over state-government regulatory documents. Phase 1 covers Mississippi only; Phase 2 extends the same architecture to seven Southeastern states across three agency types (medical, dental, real estate). The goal is to let SoS staff ask plain-English questions like "How does Tennessee's barber reciprocity compare to Mississippi's?" and get answers grounded in cited statutes and regulations — never hallucinated, with explicit inference markers when the model has to extrapolate.

How it worksQuery lifecycle (one request, end-to-end)

Following one user query — "What is the renewal fee for a Mississippi notary license?" — through the full system:

  1. Frontend calls ChatService.sendMessage() which POSTs {query, filters, history, mode} to /v2/query.
  2. Lambda (sync portion) receives the POST, creates a DynamoDB job row with status pending, then self-invokes asynchronously with InvocationType='Event' and returns 202 + job_id immediately. This is the workaround for API Gateway's hard 29-second timeout.
  3. Frontend begins polling GET /v2/query/status/{job_id} every 2 seconds (5-minute total budget).
  4. Lambda (async portion) picks up the event, calls _process_query():
    • detect_intent() classifies the query as general | comparison | reciprocity | term_count via keyword patterns.
    • resolve_model_id() maps the frontend-selected model key to a Bedrock model ID (allowlisted server-side).
    • If mode == 'chat', search is locked to the Phase 1 index and Mississippi. Otherwise it goes to the Phase 2 index with state filters.
    • search_pages() fires two concurrent OpenSearch queries: kNN against text_embedding and BM25 across nine boosted text fields. Results merged via RRF (k=60), top_k returned.
    • format_context() renders the hits as a structured plaintext context block.
    • call_bedrock_llm() invokes Bedrock Converse with the SYSTEM_PROMPT (strict grounding rules) and the assembled context.
  5. Lambda (async) writes {status: 'done', answer, citations, intent, metadata} to DynamoDB. TTL = 1 hour.
  6. Frontend's next poll returns the result. The UI renders the answer with inline citations, the grounding pill (green if all claims grounded, amber if any inferences), and the citation panel.

Key decisionsTwo execution paths (this is important)

There are two parallel codepaths in this repo that both serve queries. Knowing which one is live in production matters:

  • Path A — lambda_handler.py at the repo root (this is what's deployed). Single-file handler. Intent detection via regex/keywords. Hybrid search inline. Routes between Phase 1 and Phase 2 indexes by UI mode. The async + DynamoDB job pattern lives here.
  • Path B — backend/agents/ multi-agent orchestrator (designed but not the primary deploy target). LLM-based 9-way intent classifier in query_classifier.py. Each intent routes to a dedicated agent class (ComparisonAgent, FeeAnalysisAgent, AuthorityAgent, etc.). The ReflectionAgent for post-hoc claim verification lives here. Lambda entry: backend/agents/lambdas/orchestrator_handler.py.

DecisionPath A is what powers the demo. Path B is the full-fat architecture that the eval-improvement work would activate. They share the same OpenSearch indexes and the same hybrid-search formula, so swapping is non-breaking. The orchestrator agents are more sophisticated (per-intent prompts, structured metadata responses), but the pilot runs Path A because it's simpler to deploy and iterate during the PoC.

GotchasThings that confuse first readers

  • Two paths look like duplication. They are. Path A is the production handler; Path B is the design we'd activate for the post-hoc reflection capability and per-intent agents. Don't delete either without understanding which the live API points at.
  • Sync vs async. All real queries take the async path because Mistral L3 + multi-state context can blow past 29s. Setting {"sync": true} in the body forces sync — useful for debugging from test_lambda.py, never used by the frontend.
  • The "embedding" field name differs. Path A indexes use text_embedding (1024-dim Titan). The legacy Sagemaker code uses embedding_vector. The mismatch matters when running the orchestrator agents (Path B) against the Phase 2 index — see _search_helpers.py:62.

02Phase 1 — Mississippi Pilot

Last verified: 2026-04-20 against commit a9d7ab3

PurposeWhat Phase 1 ships

Phase 1 was the original PoC scope: a Mississippi-only RAG system over the MS Administrative Code, with a single search index, a single chat-style UI, and grounded-citation answers. Everything Phase 2 builds on — the ingestion pipeline, the Lambda handler shape, the OpenSearch index design, the trust surfaces — first shipped here.

How it worksSource documents

Documents come from the Mississippi Secretary of State's Administrative Code search portal. The crawl uses a POST-based API rather than HTML scraping — the portal exposes a JSON-style endpoint that takes an agency_id and returns a list of PDF URLs in the ACCode/ directory. Three agency IDs cover the Phase 1 scope (medical licensure, real estate commission, dental examiners). PDFs land in S3 under s3://ms-sos-legal-documents/source-documents/.

Why dual records (document + page)?

Most page records are what powers Q&A — they have the embedding, the structured extraction (statute_codes, fee_amounts, license_categories…), and the abstract text. But "term-frequency" intent ("how many times does reciprocity appear in the corpus?") needs to count occurrences across full document text, not snippet-by-snippet. Holding the full full_text on a sibling record_type=document record lets the term-count query use OpenSearch's scroll API on documents while the search query stays on pages. Both record types live in the same index, distinguished by the record_type keyword field.

DecisionThe dual-record design saves us from running two indexes for two query patterns. Cost: every document is stored twice (full text plus per-page slices). For our corpus size (~1k–5k documents per phase), the duplicated storage is negligible compared to the operational simplicity.

How it worksOpenSearch index design

Single index per phase:

  • ms-phase1-legal — Phase 1, Mississippi only.
  • multistate-phase2-legal — Phase 2, all 7 states.

Both share the same mapping (defined in ingestion/index_manager.py:26). Highlights:

  • kNN enabled with ef_search=512; the embedding field is HNSW with cosinesimil, ef_construction=512, m=16, FAISS engine, 1024 dimensions.
  • Custom legal_analyzer = standard tokenizer + lowercase + stop. Applied to raw_text, abstract_text, core_rule, compliance_requirements, testing_requirements, reciprocity_provisions, full_text.
  • fee_amounts is a nested field — required so that aggregations like "all renewal fees in Mississippi above $200" can match per-fee, not per-document. Each entry has {amount: float, fee_type: keyword, description: text, statutory_cap: float}.
  • Dates as yyyy-MM-dd||epoch_millis on effective_date and amendment_date — accepts both ISO strings and ms timestamps, important because the LLM extractor produces strings while resume code may pass timestamps.
  • Phase 2 filter fields are keywords: state, agency_type, agency_name. Used for filtering hybrid-search results by jurisdiction.

How it worksLambda handler

The deployed function is ms-sos-legal-v2 (300s timeout, 2GB memory). The handler dispatches between three modes:

  • Mode 1 — async worker. Triggered when the event has _async_job_id. Runs the heavy query, writes result to DynamoDB.
  • Mode 2 — HTTP GET. Status-check endpoint (/v2/query/status/{id}). Reads DynamoDB and returns current job state.
  • Mode 3 — HTTP POST. Default. Creates a job, dispatches an async self-invoke via the Lambda client, returns 202 + job_id.

The SYSTEM_PROMPT at lambda_handler.py:69 is the single source of truth for grounding behavior. It enforces:

  1. Every factual claim followed by inline citation in the form (Source: <filename>, p.<page>).
  2. Extrapolation wrapped in [INFERENCE]: … blocks.
  3. Final line exactly Grounding summary: <N> grounded, <M> inferred — the eval rubric checks for this verbatim.

WarningmaxTokens is capped at 2048 in the Converse call. This isn't because the model can't handle more — it's to keep the synchronous portion of the request under API Gateway's 29s timeout in the rare case someone calls with sync=true. If you raise it, also raise the API Gateway timeout or you'll get silent truncation on sync paths.

GotchasPhase 1 traps

  • Phase 1 chat mode forces state=MS even if the frontend sends no filter (see _process_query at lambda_handler.py:761). This was deliberate: chat mode = single-state Q&A; cross-state lives in research mode.
  • The compression model identifier in the Lambda handler defaults to whatever BEDROCK_MODEL_ID env var is set to. The frontend can override via the model field in the request body, but only against an allowlist (see ALLOWED_MODELS at lambda_handler.py:342). Adding a new model means updating both this allowlist and Marketplace subscriptions in AWS.
  • The Phase 1 ingestion uses a flat S3 prefix (source-documents/file.pdf); Phase 2 uses a hierarchical one (crawled-documents/TX/dental/file.pdf). The pipeline's list_pdfs() auto-detects which by counting path segments — see ingestion/pipeline.py:200.

03Phase 2 — Multi-State Extension

Last verified: 2026-04-20 against commit a9d7ab3

PurposeWhat Phase 2 adds

Phase 2 keeps the Phase 1 architecture intact but extends it along three axes:

  1. Geography. 7 states (MS, TN, AL, GA, AR, TX, LA), 3 agency types each, 21 source crawls.
  2. Extraction richness. A V2 compression schema pulls out fees, dates, license categories, testing requirements, statutory authority references, and reciprocity provisions — the metadata needed for cross-jurisdiction comparison and authority chains.
  3. Query types. A multi-agent orchestrator (in backend/agents/) supports nine intent categories beyond simple Q&A: comparison, fee analysis, term frequency, reciprocity, authority chains, etc. The deployed Lambda runs a simpler regex-based version of this dispatch.

How it worksV2 metadata model

Phase 1 used CompressedAbstract — abstract text, statute codes, compliance requirements, document type. That's enough for "answer the question" but not for "compare fees across states." Phase 2 adds CompressedAbstractV2 (extends V1) with:

agency_typeNormalized: medical, dental, real_estate
agency_nameFree-form name from the source ("Tennessee Board of Dentistry")
fee_amountsList of FeeRecord {amount, fee_type, description, statutory_cap}
effective_dateISO date when the rule took effect
amendment_dateISO date of last amendment, if mentioned
license_categoriesKeyword list (e.g. ["physician", "physician assistant"])
testing_requirementsFree text about exams, NCEES references, etc.
statutory_authority_referencesStatute codes the rule cites as its enabling authority
reciprocity_provisionsFree text about out-of-state license recognition

These fields populate via compress_v2() in Sagemaker Files/compression_agent_bedrock.py:249, driven by COMPRESSION_PROMPT_V2 at line 78 of the same file. The prompt is the dial — tuning extraction quality means editing the prompt and re-running the pipeline.

How it worksMulti-agent orchestrator (Path B)

The orchestrator (backend/agents/orchestrator.py:53) routes each query to one of nine specialized agents:

TERM_FREQUENCY"How often does X appear" — TermFrequencyAgent

Each agent has a single execute() method, takes filters explicitly, and returns a structured response with answer, citations, and an intent-specific metadata field (comparison_table, fee_table, frequency_data, reciprocity_data, authority_chain). The frontend's research view renders these as tables, charts, and chains rather than plain text.

The ReflectionAgent at backend/agents/reflection_agent.py:75 is post-hoc: it decomposes the LLM answer into atomic claims, verifies each has a supporting citation, and tags any without one. This is the architectural piece that converts self-reported grounding (today) into verified grounding (the next iteration).

How it worksHybrid search (kNN + BM25 + RRF)

This is the load-bearing retrieval call. Same formula in both lambda_handler.py:107 and backend/agents/_search_helpers.py:26. The flow:

  1. Embed the query via Titan (1024-dim, normalized).
  2. kNN query: find candidate_pool = top_k * 3 nearest vectors in the embedding field, applying the same state / agency_type filters as the BM25 side.
  3. BM25 query: bool with one match clause per text field, each with a hand-tuned boost weight (table below). minimum_should_match: 1.
  4. RRF merge: for every hit at rank r, add 1 / (60 + r + 1) to its score. A document that ranks well in both kNN and BM25 gets two contributions and rises to the top.
  5. Sort by combined RRF score, return top_k.

BM25 boost weights

Order matters: the highest-boosted field dominates when a query token appears there.

FieldBoostWhy
statute_codes4.0Statute citations are the most precise match; if the user says "73-25-29", they want that statute
statutory_authority_references3.5Same reasoning, slightly less specific (it's the authority the rule cites, not the rule itself)
section_identifier3.0"Rule §189.3", "Chapter 540-X-22" — when the user names a section, they want it
abstract_text2.0The compressed summary; high signal but not as precise as code/section
core_rule2.0The extracted single-sentence rule statement
raw_text1.5Full page text; lots of noise, lower boost
compliance_requirements1.5Extracted requirement list
testing_requirements1.5Extracted exam/cert info
reciprocity_provisions1.5Extracted reciprocity text
legal_entities1.0Named entities; lowest signal

DecisionRRF (k=60) was chosen over a tuned alpha-blend of normalized scores because RRF is parameter-free across query types. With score blending, a query where BM25 happens to return scores in the 1–5 range and kNN returns scores in the 0.7–0.95 range would systematically favor BM25 unless we re-normalized. RRF only uses ranks, so it sidesteps the calibration problem entirely. The k=60 default comes from Cormack et al. (2009) — well-tested, no reason to deviate.

How it worksIndex routing & state selection

Routing rules in _process_query (lambda_handler.py:704):

  • mode == 'chat' → ms-phase1-legal, force filter_state="MS" if no filter set, ignore filter_states.
  • mode != 'chat' (research/compare/count) → multistate-phase2-legal.
  • For multi-state intents (comparison, reciprocity): if no filter_states provided, run detect_states(query) on the query text. If still empty, default to all 7. If intent is reciprocity and MS isn't in the list, prepend it (the SoS office cares about MS reciprocity by definition).
  • Per-state result budget (multi-state queries): if ≤2 states, 6 results each; ≤4 states, 4 each; ≥5 states, 3 each. This caps total context tokens to keep Mistral latency under budget.

GotchasPhase 2 traps

  • Field-name drift between Path A and Path B. Path A's search_pages uses text_embedding; Path B's hybrid_search uses embedding_vector. They were written against different index versions. If you swap orchestrator to point at the Phase 2 index, you need to align field names or duplicate the embedding under both names.
  • State inference is greedy. detect_states() at lambda_handler.py:394 scans for substring matches like " al ", " ar ". Mentioning Alabama in a query like "I have an Alabama license" correctly tags AL — but a query about "Algeria" would also match " al " in some surrounding text. We've never been bitten by this in practice with regulatory queries; flag if it shows up.
  • Per-state budget is hard-coded. Tuning it is a one-line change in _process_query but easy to miss when latency profiling. Production tuning would derive these from observed Mistral p95 latency, not eyeball them.

04Web Crawling

Last verified: 2026-04-20 against commit a9d7ab3

PurposeWhat the crawlers do

Each state publishes its administrative regulations differently — Mississippi uses a POST-based search API, Tennessee publishes flat HTML directories of PDFs, Alabama serves chapter-by-chapter PDFs from admincode.alabama.gov, etc. The crawler subsystem normalizes all of this into a single output: PDFs on disk, organized as crawled_documents/<STATE>/<agency_type>/file.pdf, plus a manifest.json tracking what came from where. From there, the ingestion pipeline takes over.

How it works21 targets

A CrawlTarget dataclass (backend/crawlers/config.py:16) describes one (state × agency_type) pair: state code, agency type (medical | dental | real_estate), agency name, source URL, crawler type (which subclass to use), allowed domains, and an extra dict for crawler-specific config (agency_id for MS, chapter list for AL, etc.). The full set lives in module-level lists (MS_TARGETS, TN_TARGETS, …) and totals 21 entries.

Lookups go through the registry:

get_target(state, type)Single target, e.g. ("TN", "dental")
get_targets_for_state(state)All 3 agency types for a state
get_all_targets(states=, types=)Filtered subset of all 21

How it worksManifest & diff detection

After every crawl run, results are aggregated into a CrawlManifest (backend/crawlers/manifest.py:45): per-target timestamps, file hashes, sizes, source URLs. This is the substrate for the "weekly source refresh loop" advertised in the demo: crawl → diff → re-ingest only what changed → notify a human. The crawler tier is built today; the orchestration loop that wraps it is designed but activates on production deployment.

GotchasCrawler traps

  • max_subpages = 20 is a hard cap. If a regulator's index page links to 40 chapter PDFs, the crawler will only follow the first 20 keyword-matching subpages. Bumping the cap means more crawl time and more politeness pressure on the source. Tune per-target via constructor arg, not globally.
  • Filename collisions are auto-renamed. If a state publishes two files named Chapter1.pdf from different subpages, BaseCrawler.download_file() at line 258 appends _2, _3 automatically. The manifest preserves the original URLs so you can trace the rename.
  • No retry on download failures. The fetch() method retries (2 retries, exponential backoff). The download_file() path does not — a transient 503 mid-stream loses the file. The crawl manifest records the error; re-running the crawl re-attempts.
  • The crawler is not on a schedule today. Crawls are kicked off manually via the CLI. The "weekly auto-refresh" in the demo deck is the architecture being ready, not a cron job that ships in the PoC.

How it worksExtraction pipeline (per-PDF)

IngestionPipeline._process_document() at pipeline.py:329 is the canonical per-PDF flow. Annotated:

  1. Generate doc_id as the first 16 hex chars of sha256(state/agency/filename). This guarantees uniqueness across states even if filenames collide ("rules.pdf" exists in three states).
  2. Download PDF from S3 to /tmp/sos-ingest-v2/<filename>.
  3. PyMuPDF extracts text per page. Pages with empty text are scanned PDFs.
  4. Textract OCR loop: for each empty-text page, render to PNG at 200 DPI and call detect_document_text. Cost: ~$0.0015/page.
  5. Table-extraction loop: is_table_page() heuristic (≥3 dollar signs, ≥3 fee keywords, ≥60% short lines, column alignment) gates the call. Triggered pages get analyze_document with TABLES feature; the table is reconstructed and appended to the page text. Cost: ~$0.015/page.
  6. Build document record from concatenated full text (with [Page N] markers for the term-frequency scroll).
  7. Per-page processing: low-value pages (is_low_value_page() — text < 80 chars or < 30 alphabetic) skip the LLM and get raw text only. The rest go in batches of 3.
  8. Mistral batched extraction: call_mistral_batch() sends 3 pages per call (cost ~$0.012 vs $0.015 for 3 individual calls). Fallback to per-page on parse failure. Each page gets back the structured V2 fields.
  9. Per-page Titan embedding: create_embedding_text() builds the embedding input (abstract + core_rule + statute_codes + section_identifier + raw_text[:2000], joined by |); get_text_embedding() calls Titan v2 (1024-dim normalized).
  10. Bulk index via IndexManager.bulk_index(): 50 records per chunk; if a chunk fails with 413 (too large), fall back to one-record-at-a-time.

Why batch size 3?

BATCH_SIZE = 3 at pipeline.py:109. Trade-off: batches of 1 are most reliable (single page, clean prompt) but most expensive (one API call per page). Batches of 5+ start hitting Mistral's max_tokens cap because a 5-page batch can produce 7500+ tokens of structured output. Three is the empirical sweet spot — ~3x throughput vs single-page, no truncation in practice. Each batch costs ~$0.012 (vs $0.005 × 3 = $0.015 for individual calls — modest savings on API cost, larger savings on wall time).

How it worksLease ledger (multi-session coordination)

The pipeline can be run from multiple terminals or laptops simultaneously without each session re-processing the same documents. The mechanism is a file-locked JSON ledger at ingestion_ledger_<index>.json (ingestion/lease.py):

  • Session ID: defaults to <hostname>-<pid>-<short uuid>; can be set with --session-id.
  • claim_batch(candidate_keys, batch_size): under flock, looks at all unclaimed/uncompleted/unfailed keys, claims up to batch_size for the current session, writes back, releases lock. Returns the claimed list.
  • heartbeat() updates the lease's heartbeat_at timestamp; called between batches.
  • Stale-lease reaping: any lease whose heartbeat is older than 600 seconds (10 minutes) is considered abandoned. The next claim_batch() call returns its keys to the unclaimed pool.
  • mark_done(key) / mark_failed(key): move the key out of the lease into the global completed/failed sets.
  • release() on shutdown: returns any unfinished claims to unclaimed.

This pattern requires no DynamoDB, no Redis — just a shared filesystem. Works fine for the team's "two laptops at the same desk" scale; doesn't scale to many concurrent EC2 workers (you'd want DynamoDB conditional writes for that).

How it worksSSO refresh & resume

SSO tokens expire after one hour. A multi-hour ingestion would die otherwise. AWSSession.ensure_valid() at aws_session.py:91 calls STS GetCallerIdentity to test; on failure, prompts a browser SSO re-login (or exits cleanly with instructions for managed shells).

The pipeline calls ensure_valid() every 10 documents (CHECK_CREDS_EVERY_N_DOCS) and re-creates the boto3 clients on refresh. If indexing fails mid-batch with a token-expired error, _process_document() catches and retries once after a fresh refresh (pipeline.py:524).

Resume is automatic: the progress file is per-index (ingestion_progress_<index>.json). Re-running the pipeline picks up where the last run left off via the lease ledger; the legacy progress file is folded in once on startup for backward compatibility.

GotchasIngestion traps

  • The LLM occasionally returns "Up to" or "varies" for fee amounts. Caught by _sanitize_fees() which extracts numbers via regex or drops the entry. If the value is non-numeric and has no embedded number, the fee is dropped silently — affects fee-comparison eval scores. Worth surfacing the drop count in pipeline logs.
  • The pipeline is single-process per session. Parallelism comes from running multiple sessions simultaneously, not from threading inside one. CPU-bound stages (PDF parsing) and I/O-bound stages (Bedrock, OpenSearch) all run sequentially per document. Acceptable for the corpus size; a redesign would be needed for larger volumes.
  • Textract is the most expensive step — $0.015 per table page can dominate cost on fee-heavy documents. The is_table_page() heuristic is conservative (score ≥ 3) but does fire false positives on heavily-formatted text. False positives waste money but don't degrade results.
  • The legacy progress file is read once and migrated. If you delete it, no harm done (already in the ledger). If you have a stale one with a wildly different set of completed_keys than the ledger, the merge can mark already-failed docs as completed. Use --reset to nuke and re-run from scratch.

07Evaluation Framework

Last verified: 2026-04-20 against commit a9d7ab3

PurposeWhy we built this

We needed a way to measure whether the system is doing what we claim — specifically, whether it's honestly grounding its answers. "Did it sound right?" is not a measurement; "did it cite a real source for every factual claim?" is. The eval is a 25-question pilot scored on four axes by an LLM judge.

How it works4-axis rubric

Every question gets a 0.0–1.0 score on each axis. The judge prompt at evals/judge.py:28 gives explicit grading criteria for each:

AxisWhat it measuresPilot score
Groundedness Every non-inference factual claim has a citation 0.66
Inference Honesty Extrapolations marked with [INFERENCE]:, Grounding summary: line present and accurate 0.80
Correctness Substantive answer matches the ideal answer 0.56
Jurisdiction Cited statutes/rules are from the correct state 0.88

Two derived headlines:

  • Primary score = mean(Groundedness, Inference Honesty) = 0.73. This is the "do we trust the answer?" headline because grounding × honesty tells you whether the system is straight with you about what it knows.
  • Overall = mean(all 4 axes) = 0.72.

Judge model

The judge runs on Amazon Nova Pro (us.amazon.nova-pro-v1:0), set via the JUDGE_MODEL_ID env var (evals/judge.py:28). Why Nova: it's available on our account without Marketplace subscription gymnastics, fast enough for 25 sequential calls, and cheap. Switching to a different judge model (e.g., Claude Sonnet if access opens) is a single env-var change.

CaveatThe judge is the same model family as the system under test (both are Amazon-hosted via Bedrock). Standard LLM-as-judge concern: model-family bias. The 4-axis rubric mitigates this by being prescriptive (the judge isn't asked "is this good?" — it's asked "does the answer end with a Grounding summary: line?"). A counsel-validated 80-question set with human spot-checks is the next reliability step.

How it worksPilot results

Per-pattern breakdown — questions are tagged by query pattern so we can see where the system shines and where it struggles:

Worst questions (and what they reveal)

  • q-017 — infrastructure failure (list index out of range in the eval harness). Excluded from the 24-question primary average. Harness bug, not model failure.
  • q-010 (Arkansas physician academic license renewal) and q-014 (Alabama/Tennessee real estate Recovery Fund fees) both scored 0.00 across all axes. Common cause: the model returned an answer without the required Grounding summary: line, which the rubric treats as a complete failure. Pattern correlates with synthesis-heavy fee-comparison questions where the model is doing too much work and forgets the closing structure.

This is the proximate cause of the fee_comparison 0.52. The fix isn't more training — it's deterministic post-hoc verification (the ReflectionAgent design) that doesn't trust the model to self-mark.

Key decisionsMethodology choices

  • N=25, AI-curated. The golden set was assembled by Claude against public state-government sources, not validated by counsel. Real production-quality eval would be ~80 questions reviewed by Colby. The 25-question pilot validates the methodology and catches gross failures; the score numbers are indicative, not authoritative.
  • Single run, no variance analysis. Each question runs once. We don't know how much score-jitter there is across runs at temperature 0.1. Easy to add (run each question 3x, report mean ± stdev), deferred for the pilot.
  • Self-reported grounding. The Grounding summary: count is whatever the model claims it is. The judge checks the line is present and parseable but doesn't independently verify the count by reading the answer. Reflection-agent verification is the architectural answer to this.
  • 4 axes, not 1. A single composite score would let "looks correct, isn't grounded" pass. Splitting into 4 surfaces the trade-offs and tells us where to invest.

GotchasEval traps

  • The eval runs against the live Lambda. Each run consumes Bedrock budget proportional to N × (system tokens + judge tokens). 25 questions ≈ ~$3 per full run. Don't loop the eval — set it intentionally.
  • The poll timeout matters. run_eval.py:57 has a MAX_POLL_S budget per question. If a question takes longer than that, the eval records a failure even if Lambda eventually completes. Tune carefully when adding heavy questions.

  • OpenSearch nested fields require strict types. A single bad fee value can fail the whole document index without sanitization.
  • Filename collisions across states are real (rules.pdf exists in 5 states). The doc_id hash includes state and agency to disambiguate.
  • Textract sync APIs cap at 5MB image input. 200 DPI for full-page PNGs is calibrated to stay under.

Frontend

  • Vite env vars must start with VITE_. Forgetting the prefix → undefined at build time, no warning.
  • Polling loop doesn't cancel on unmount. Wasted API calls until 5min timeout.
  • localStorage chat history grows unbounded. Manual clear required for long sessions.
  • Tier chips deliberately avoid percentages — never validated. Don't add a "% MATCH" label.

Eval

  • Each full eval run costs ~$3 in Bedrock. Don't loop it.
  • Judge model bias: Nova judging Mistral; same model family caveat applies.
  • Self-reported grounding count. The Grounding summary: N is whatever the model claims — verification is post-hoc.
  • N=25 single-run, no variance. Score numbers indicative, not authoritative.
